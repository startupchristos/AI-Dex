# Clean Markdown

**Single source of truth for markdown cleanup.** Skills and commands that need to normalize dirty markdown (Pandoc artifacts, Word exports, paste-from-doc) should reference this command. Normalize a markdown file that was generated by Pandoc (`pandoc --from docx --to markdown`) or similar into clean, standard markdown.

---

## Invocation Modes

**Interactive mode** (default): User runs this command directly. Full flow: gather input, apply cleanup, show diff, show cleaned text, confirm output destination, write.

**Programmatic mode**: Skill or command invokes this to clean text in memory. Apply Step 2 (cleanup rules) only; return cleaned text to caller. No diff, no prompts, no file I/O. Use when the caller instructs "apply util-clean-markdown in programmatic mode" or "clean this text and return it" or similar.

**How to detect**:
- **Interactive**: User explicitly ran this command (e.g. `/util-clean-markdown`, "clean this markdown file", "run util-clean-markdown on X").
- **Programmatic**: You are mid-workflow in another skill/command that gave you text to clean and instructed you to return the result.

---

## Interactive Mode

### Step 0: Gather Input

Accept one of:
- **File path** — User provides path to a `.md` file
- **Pasted text** — User pastes markdown into the conversation

If neither is provided, ask.

Read the file (if path) or use the pasted text. Record the source (file path or "pasted") for later steps.

### Step 1: Apply Cleanup Rules

Apply all transformations from Step 2 (Cleanup Rules) below. Track counts for the diff summary.

### Step 2: Cleanup Rules (Shared — apply in order)

Apply all of the following transformations in order:

#### 2a. Grid Tables -> Pipe Tables

Pandoc grid tables look like:

```
+----------+----------+
| Header 1 | Header 2 |
+==========+==========+
| Cell 1   | Cell 2   |
+----------+----------+
| Cell 3   | Cell 4   |
+----------+----------+
```

Convert each grid table to a standard pipe table:

```
| Header 1 | Header 2 |
|----------|----------|
| Cell 1   | Cell 2   |
| Cell 3   | Cell 4   |
```

Rules:
- The row with `=` separators is the header separator row -- the rows above it are headers, rows below are data.
- Strip leading/trailing whitespace from each cell value.
- Use `|---|` style separators (dashes only, no colons unless alignment was explicitly set).
- If a grid table has no `=` row, treat the first row as the header.

#### 2b. Strip Image Size Attributes

Remove `{width="Xin" height="Yin"}` and similar attribute blocks after image tags.

Before: `![Alt text](image.png){width="3.5in" height="2in"}`
After:  `![Alt text](image.png)`

Also strip any other curly-brace attribute blocks on image lines (e.g. `{.class}`, `{#id}`).

#### 2c. Convert Pandoc Underline Spans to HTML

Pandoc uses `[text]{.underline}` to mark text that should be underlined. Many markdown renderers ignore this attribute. Convert to semantic HTML so underline actually renders:

- **Pattern:** `[X]{.underline}` where X is the span content (no unescaped `]` inside)
- **Replacement:** `<u>X</u>`
- **Scope:** Only spans with exactly `{.underline}` as the attribute

Examples:
- `[Termination]{.underline}` → `<u>Termination</u>`
- `**[Relationship of Parties]{.underline}:**` → `**<u>Relationship of Parties</u>:**`
- `[25 SE 2nd Ave Ste 550, Miami, FL 33131]{.underline}` → `<u>25 SE 2nd Ave Ste 550, Miami, FL 33131</u>`

Implementation: Replace each match of `\[([^\]]*)\]\{\.underline\}` with `<u>$1</u>`.

#### 2c2. Strip Pandoc Attribute Blocks

Pandoc adds attribute blocks (e.g. `{.mark}`, `{.TOC-Heading}`, `{#id}`, `{#id .class}`) that standard markdown renderers show as literal text. Remove them.

**Span with .mark inside a link (apply first):** `[[X]{.mark}](#anchor)` → `[X](#anchor)`

Pattern: `\[\[([^\]]*)\]\{\.mark\}\]\(([^)]*)\)` → `[$1]($2)`

Example:
- `[[2. Market Analysis]{.mark}](#market-analysis)` → `[2. Market Analysis](#market-analysis)`

**Bare span with .mark:** `[X]{.mark}` → `X` (extract content, remove brackets and attribute)

Pattern: `\[([^\]]*)\]\{\.mark\}` → `$1`

Example:
- `[2. Market Analysis]{.mark}` → `2. Market Analysis`

**Attribute blocks on headings/blocks:** Remove `{#id}`, `{.class}`, `{#id .class}` and similar. Match `\{[#.][^}]*\}` and replace with empty string.

Examples:
- `# Table of Contents {#table-of-contents .TOC-Heading}` → `# Table of Contents`
- `[]{#_Toc186198492 .anchor}` → `` (remove entirely)

**Order:** Apply after 2c (underline). Handle `.mark`-inside-link before bare `.mark`. Do not strip `{.underline}` (handled by 2c).

#### 2c3. Strip Pandoc TOC Page-Number Links

Pandoc encodes Word TOC page numbers as a nested link appended to each TOC entry:

```
[Section Title](#anchor) [[22](#anchor)](#anchor)
```

The trailing `[[N](#anchor)](#anchor)` is the page number artifact. Strip it entirely.

**Pattern:** ` \[\[[0-9]+\]\(#[^\)]+\)\]\(#[^\)]+\)` → `` (remove, including the leading space)

Examples:
- `[1. Introduction](#introduction) [[3](#introduction)](#introduction)` → `[1. Introduction](#introduction)`
- `[2.1. Market Profile](#market-profile) [[8](#market-profile)](#market-profile)` → `[2.1. Market Profile](#market-profile)`

**Order:** Apply after 2c2 (so any `.mark` spans in the link text are already resolved).

#### 2c4. Fix Ordered-List Marker Pattern in Link Labels

Some markdown parsers (including VS Code markdown-it) misparse link labels that begin with an ordered-list marker pattern (`N. `). When a link label starts with `[N1. N2` (a number, period, space, followed by another number), the `N1. ` is mis-identified as an ordered list item marker, breaking the link rendering entirely.

**Pattern:** `\[(\d+)\. (\d+)` → `[$1.$2` (collapse the space between the two number parts)

**Scope:** Only fires when BOTH sides of the `. ` are digits -- leaves entries like `[1. Introduction]` untouched.

Examples:
- `[4. 2 Competitor Registration](#competitor-registration)` → `[4.2 Competitor Registration](#competitor-registration)`
- `[4. 5 SWOT Analysis of Competitors](#swot-analysis)` → `[4.5 SWOT Analysis of Competitors](#swot-analysis)`

**Also fix the matching heading text** when present in the document body (e.g. `## 4. 2 Competitor Registration` → `## 4.2 Competitor Registration`), so the heading text stays consistent with the link label.

**Order:** Apply after 2c3. Apply globally (not just in TOC sections) since headings in the body have the same pattern.

#### 2c5. Escape List-Marker Period in Link Labels

Some markdown parsers (including VS Code markdown-it) interpret link text that starts with `N. ` (number, period, space) as an ordered list item, breaking the link and causing it to render across multiple lines. Escape the period so it is treated as literal text.

**Pattern:** `\[(\d+)\. ` → `[$1\. ` (escape the period when link label starts with digits, period, space)

**Scope:** Only when the character after the space is not a digit (otherwise 2c4 handles it). Targets `[1. Introduction]`, `[2. Market Analysis]`, etc., but not `[4.2 ...]` or `[1.1 ...]` (no space after the first period).

Examples:
- `[1. Introduction](#introduction)` → `[1\. Introduction](#introduction)`
- `[2. Market Analysis](#market-analysis)` → `[2\. Market Analysis](#market-analysis)`
- `[8. Conclusions and Recommendations](#conclusions)` → `[8\. Conclusions and Recommendations](#conclusions)`

**Order:** Apply after 2c4. The escaped period is consumed during parsing; rendered output still shows "1. Introduction".

#### 2d. Unescape Backslash Escapes

Pandoc over-escapes many characters. Unescape the following:

| Escaped | Plain |
|---------|-------|
| `\'`    | `'`   |
| `\"`    | `"`   |
| `\#`    | `#`   |
| `\<`    | `<`   |
| `\>`    | `>`   |
| `\%`    | `%`   |
| `\&`    | `&`   |
| `\@`    | `@`   |

Do NOT unescape characters that are legitimately escaped in markdown (e.g. `\*`, `\_`, `\[`, `\]`, `\(`, `\)`, `\`` ).

#### 2e. Join Mid-Paragraph Line Breaks

Pandoc wraps long lines at ~72 characters, inserting hard line breaks in the middle of paragraphs. Join these back into single lines.

Rules:
- A blank line (empty or whitespace-only) marks a paragraph boundary -- preserve it.
- A line ending without two trailing spaces that is followed by a non-blank line within the same paragraph should be joined with a single space.
- Do NOT join lines that are:
  - Headings (`#`, `##`, etc.)
  - List items (`- `, `* `, `1. `, etc.)
  - Code blocks (indented 4 spaces or inside ``` fences)
  - Table rows (lines starting with `|` or `+`)
  - Blockquotes (`>`)
  - Horizontal rules (`---`, `***`, `___`)
  - YAML frontmatter (between `---` delimiters at top of file)

#### 2f. Remove Trailing Whitespace

Strip trailing spaces and tabs from every line.

#### 2g. Remove Zero-Width and Invisible Characters

Remove these Unicode characters that cause rendering issues:
- Zero-width space (U+200B)
- Zero-width non-joiner (U+200C)
- Zero-width joiner (U+200D)

#### 2h. Replace Non-Breaking Spaces

Replace non-breaking space (U+00A0) with a regular space. Non-breaking spaces can break layout or cause copy-paste issues.

#### 2i. Collapse Multiple Blank Lines

Reduce multiple consecutive blank lines (empty or whitespace-only) to at most 1 blank line between blocks. Preserve paragraph boundaries; only collapse excess blanks.

#### 2j. Convert Manual Numbering to Native Lists

Pandoc or Word exports may produce manual numbering that doesn't render as proper lists. Convert to native ordered list syntax:
- `1.1.`, `1.a.`, `1.1.1.` (as plain text or under bullets) → native ordered list with indentation
- Top level: `1.` `2.` at line start
- Sub-level: indent 4 spaces + `1.` `2.`
- Third level: indent 8 spaces + `1.` `2.`

Example: `* 1.1. Left section` or `1.a. Left section` → `    1. Left section` (4 spaces before `1.`)

#### 2k. Replace Em-Dashes and En-Dashes

Replace em-dash (—, U+2014) and en-dash (–, U+2013) with appropriate alternatives per Dex style:
- Use hyphen `-` for ranges (e.g. "2020-2024")
- Use comma, period, colon, semicolon, or restructure for parenthetical or break-in-thought usage
- No em-dashes or en-dashes in cleaned output

### Step 3: Show Diff Summary and Cleaned Text

Display:

```
Cleanup complete. Changes applied:

- Grid tables converted:        N
- Image attributes stripped:    N
- Pandoc underline spans converted: N
- Pandoc attribute blocks stripped: N
- TOC page-number links stripped: N
- Ordered-list marker patterns in links fixed: N
- List-marker period in link labels escaped: N
- Backslash escapes fixed:      N
- Paragraph lines joined:       N
- Trailing whitespace fixed:    N
- Zero-width chars removed:     N
- Non-breaking spaces fixed:    N
- Blank lines collapsed:        N
- Manual numbering converted:   N
- Em/en-dashes replaced:        N
```

If no changes were found in a category, show `0` (do not omit the row).

Display the cleaned text in a code block so the user can copy-paste.

If no changes are needed overall, say: "File looks clean — no Pandoc artifacts found." and stop.

### Step 4: Offer to Save

Ask: "Save to file? [In-place] [Save as ...-clean.md] [Other path] [No]"

- **In-place**: Overwrite the source file. Only offer when input was a file path.
- **Save as ...-clean.md**: Save to same directory with `-clean` suffix (e.g. `notes.md` → `notes-clean.md`). Offer for both file and pasted input.
- **Other path**: User specifies a path.
- **No**: Do not write; user keeps the copy from the code block above.

### Step 5: Write if Confirmed

If the user chooses In-place, Save as, or Other path, write the cleaned content to the specified location using the Write tool.

Confirm: "Done. `<filename>` has been saved."

If the user chooses No, confirm: "No file saved. Use the cleaned text from above."

## Programmatic Mode

When a skill or command invokes util-clean-markdown to clean text in memory:

1. **Input**: Text provided by the caller (from file, paste, or prior step).
2. **Apply**: Step 2 (Cleanup Rules) only. No diff, no prompts.
3. **Return**: Cleaned text to the caller. The caller decides what to do with it.
4. **On error**: Return an error indicator (e.g. `ERROR: <description>`) so the caller can handle it. Do not throw; return a structured failure the caller can detect.

---

## Error Handling

**Interactive mode:**
- If the file does not exist, say so and stop.
- If the file is not a `.md` file, warn the user and ask if they want to proceed anyway.
- If no changes are needed, say: "File looks clean — no Pandoc artifacts found." and do not prompt to write.

**Programmatic mode:**
- On failure (e.g. invalid input, empty content): Return `ERROR: <reason>` instead of cleaned text so the caller can handle it.
